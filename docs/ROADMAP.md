# Drop — 로드맵 & 설계 노트

## 현재 상태

**1단계(MVP) — 완료.** 파일 업로드(드래그앤드롭 + 안드로이드 공유 시트), 다운로드, 소프트 삭제
휴지통(30일 자동 정리), 최초 관리자 부트스트랩 + 로그인, PWA 설치, Docker Compose + Caddy 배포,
Proxmox LXC 원클릭 설치.

**추가로 구현된 것** (원래 1단계 체크리스트를 넘어서지만, 실사용하면서 필요해져서 추가한 것들):

- 청크 업로드(8MB 단위) — 대용량 파일도 요청 하나당 메모리 사용량이 일정하고, 앱이 중간에
  죽어도(모바일 백그라운드 종료 등) 같은 파일을 다시 선택하면 서버에 남은 진행 상황부터
  이어 올린다 (`apps/web/lib/pendingUploads.ts`).
- 여러 파일을 체크박스로 선택해 zip으로 일괄 다운로드 — 브라우저의 "다중 자동 다운로드 차단"을
  피하려고 서버에서 하나로 묶어 보낸다.
- 다중 계층 폴더 구조, 파일 이동. 업로드는 항상 루트로 들어오고 나중에 폴더로 옮기는 방식.
- 클릭 시 미리보기(이미지/동영상/오디오/PDF/일반 텍스트) — 어떤 mimeType을 inline으로 보여줄지
  API/웹이 공유하는 정책 함수(`packages/shared/src/previewPolicy.ts`) 하나로 결정한다.
  `image/svg+xml`과 `text/html` 등은 인라인 스크립트 실행(XSS) 위험 때문에 의도적으로 제외.
- 하단 네비게이션(홈·파일·업로드·휴지통·더보기) — stash의 바텀 네비 스타일을 그대로 재사용.
- 같은 이름으로 업로드하면 덮어쓰지 않고 자동으로 "이름 (2).확장자" 식으로 별개 파일을 만든다
  (`apps/api/src/lib/uniqueName.ts`).

## 알려진 제약사항

- **삼성 인터넷 브라우저의 공유 시트 버그.** 삼성 인터넷으로 카카오톡/갤러리 등에서 파일을
  공유하면 업로드가 `"Part terminated early due to unexpected end of multipart data"`로
  실패한다. 실제 배포(도메인 + Cloudflare Tunnel)에서 재현 로그를 확인한 결과, 파일 크기는
  1.8MB로 전혀 크지 않았고 카카오톡/갤러리 등 원본 앱과 무관하게 항상 재현됐다 — 공유를
  처리하는 브라우저(삼성 인터넷)가 `Content-Length`를 실제 전송량보다 적게 잘못 계산해서
  보내는 것으로 보인다. Node의 HTTP 서버는 선언된 Content-Length만큼만 읽고 요청을 끝난 걸로
  간주하므로, 진짜 멀티파트 종료 경계가 오기 전에 스트림이 끊긴다. **서버에서 고칠 수 있는
  종류의 문제가 아니다** (선언된 길이 이후의 바이트는 애초에 서버에 전달되지 않는다). Chrome으로
  공유하거나, 앱을 직접 열어 업로드 탭에서 파일을 선택하면 우회된다(그 경로는 멀티파트가 아니라
  청크 업로드라 이 버그를 타지 않는다).
- **폴더로 이동 시 이름 충돌 미처리.** 업로드 시점에는 같은 이름이 있으면 자동으로 "(2)"를
  붙이지만(`uniqueName`), 파일을 폴더로 **이동**시킬 때 그 폴더에 이미 같은 이름이 있으면
  처리하지 않는다. 같은 유틸(`apps/api/src/lib/uniqueName.ts`)을 이동 라우트에도 적용하면
  쉽게 추가할 수 있다.
- **폴더 이름 자체의 중복 미처리.** 같은 부모 폴더 안에 폴더 이름이 겹쳐도 막지 않는다.
- **백업/복구 전략 없음.** Postgres 데이터와 업로드 파일(`uploads` Docker volume)을 백업하는
  절차가 문서화/자동화되어 있지 않다.
- **로그인 외 API에 rate limit 없음.** 실제 도메인으로 노출된 상태에서 업로드/다운로드
  엔드포인트에 별도 남용 방지가 없다. 개인/가족용 규모라 지금은 우선순위를 낮게 봄.
- **iOS 미지원 (설계상 의도된 제약).** Web Share Target API(공유 받는 쪽)를 iOS Safari가
  지원하지 않아서, iOS에서는 공유 시트 업로드가 원천적으로 불가능하다. 웹앱을 직접 열어서
  업로드하거나 iOS 단축어(Shortcuts)로 우회해야 한다.
